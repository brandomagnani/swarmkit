Maker's Schedule, Manager's Schedule (2009)

9/10 retrospective score
Summary

Paul Graham's influential essay argues that people who make things (programmers, writers, designers) operate on a fundamentally different schedule than managers. Makers need large uninterrupted blocks of time (ideally half-days or full days) to do deep work, while managers work in hour-long appointment blocks. The central complaint: a single meeting can destroy a maker's entire afternoon by fragmenting it into pieces too small for meaningful work. The discussion thread explores practical implementations of this principle, with commenters sharing their experiences scheduling maker time, debating whether the theory is actionable in practice, and discussing enterprise dysfunction. Notable themes include memory allocation analogies, block scheduling strategies (dedicating specific days to maker time), the tension between maker and manager roles, and skepticism about whether better meetings could solve the real underlying problem.

What Happened

The concept became widely influential in tech culture and was adopted as a principle by many engineering organizations. However, the practical outcome has been mixed. Remote work during the COVID-19 pandemic (2020) initially seemed like it could enable maker schedules—work-from-home removed commute and provided more control over one's time. Yet the opposite occurred: meeting frequency increased by 69.7% post-2020, and the number of meetings has become a chronic problem in tech. Major companies like Dropbox institutionalized 'Hack Week' (dedicated uninterrupted time), and Google added 'Focus Time' features to Calendar. However, these remain aspirational: highly productive engineering teams achieve 70% maker time, but many organizations fall far short. Google's own '20% time' program, which embodied similar thinking, never reached more than 10% actual participation as quarterly pressures grew. The article's diagnosis was correct, but the industry's ability to implement the solution has remained limited. Remote work eliminated commuting time but replaced it with more asynchronous interruptions via Slack, email, and expanded meeting culture. The pandemic actually worsened the problem despite theoretical conditions being favorable for maker time.

Announcing TypeScript 1.7

9/10 retrospective score
Summary

In November 2015, Microsoft announced TypeScript 1.7, highlighting major improvements including polymorphic `this` typing, async/await support, and continued performance improvements. The discussion reflected the JavaScript community's evolving stance on TypeScript—a Microsoft-created superset of JavaScript that provides optional static typing. Commenters were largely positive about TypeScript's incremental typing model and its integration with editors like Visual Studio Code, though some raised concerns about Microsoft fragmenting JavaScript standards rather than contributing to the language itself. Others argued over TypeScript vs Flow (Facebook's competing type checker), debated whether explicit typing was necessary overhead, and discussed the architectural question of whether TypeScript should separate concerns by using Babel for transpilation.

What Happened

TypeScript decisively won the JavaScript type-checking war. By 2024-2025, TypeScript adoption grew from ~12% in 2017 to 35-43.6% of developers actively using it, with 67% of TypeScript users writing more TypeScript than JavaScript. Over 4.2 million public repositories now use TypeScript. Flow, Facebook's competing type checker, failed to gain significant traction and essentially lost the standards battle—it remained a niche tool and Meta/Facebook later downplayed its use. Microsoft's decision to keep TypeScript as a unified compiler (rather than separating transpilation concerns to Babel) proved commercially successful, and the ecosystem rallied around TypeScript. Over 70% of new React projects now use TypeScript. TypeScript became embedded in frameworks (Angular, Next.js, NestJS) and 90% of Fortune 500 companies adopted or are transitioning to TypeScript. The concern about fragmentation proved unfounded—JavaScript evolved to adopt many features TypeScript pioneered (decorators, async/await patterns), and TypeScript's existence actually influenced JavaScript standards development. The debate about whether developers should use types proved decisively in favor of types—lack of type safety is now cited by 32% of developers as JavaScript's biggest struggle.
